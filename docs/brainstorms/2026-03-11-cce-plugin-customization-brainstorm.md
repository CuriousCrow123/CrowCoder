# What Custom Agents, Skills, and Modifications Would Make CCE More Effective for CrowCoder?

## What We're Building

A set of project-specific customizations to the `cce` plugin fork (17 agents, 35 skills) that close the gaps between its generic compound-engineering tooling and the CrowCoder project's specific stack (Astro 5 + Svelte 5 runes + Tailwind v4 + Zod v4) and architecture (island hydration, cross-island shared state, dev-time parameter tuning, SM-2 spaced repetition, popup state machine).

## Why This Approach

The fork was created specifically to be customized. The original compound-engineering plugin is optimized for Rails/Ruby/Every.com workflows. After trimming 11 agents and 12 skills in Phase 4, the remaining tooling is good but generic. These customizations make the review and work loops aware of the actual patterns and risks in this project.

## Key Decisions

### Priority Order: Batch by Priority

Do the two high-priority agents first, then everything else. This front-loads the hardest and most valuable work.

### Execution Order

**Batch 1 — High Priority Agents:**

1. **Svelte 5 Races Reviewer** (fork `julik-frontend-races-reviewer`)
   - Replace Stimulus/React/Hotwire idioms with Svelte 5 equivalents
   - Keep the core race-detection philosophy (state machines, timer cancellation, CSS transition races, Promise cleanup)
   - Add Svelte 5-specific race patterns:
     - `$effect` without cleanup return (leaked subscriptions/timers)
     - `$state` object mutations that don't trigger reactivity
     - Cross-island race: two Astro islands reading same `.svelte.ts` singleton, hydrating at different times
     - `transitionend` never firing if element removed mid-transition (popup state machine risk)
     - `$effect` running during SSR (silently no-ops, masks bugs)
   - Keep the witty, direct review style ("Julik" persona)

2. **Astro Island Architecture Reviewer** (new agent)
   - Hydration directive correctness (`client:load` vs `client:visible` vs `client:idle`)
   - SSR-incompatible code in `.astro` files (browser APIs, `$effect` in server context)
   - Shared state contract: `.svelte.ts` module singletons must be the only cross-island communication mechanism
   - Vite module deduplication assumptions
   - Inline `<script>` tags generated by Astro island hydration — may conflict with strict CSP policies on GitHub Pages

**Batch 2 — Quick Wins:**

3. **Fix Ruby examples in `/cce:plan` template** — Replace `app/services/user_service.rb:42` and similar with TypeScript/Astro/Svelte equivalents
4. **Context7 framework hints** — Add instructions to `framework-docs-researcher` and `best-practices-researcher` to use Context7 MCP for Astro 5 and Svelte 5 docs
5. **Zod discipline rules** — Augment `kieran-typescript-reviewer` with a new section: no dual-maintained interfaces, `.parse()` at all boundaries, schema versioning for localStorage

**Batch 3 — Medium Priority:**

6. **Dev-gate checker skill** — Runs `npm run build` and scans output bundle for tuning-system identifiers (`ParamPanel`, `GlobalParamPanel`, `__tunable`, `params-writer`)
7. **A11y reviewer agent** — ARIA live regions, focus traps in popups, keyboard navigation, color contrast
8. **SM-2 domain knowledge agent** (research type, not review) — Spaced repetition edge cases: first review vs subsequent, quality score mapping, interval overflow, testing time-dependent algorithms. Consulted during implementation rather than invoked during code review.

**Batch 4 — Workflow:**

9. **Param sidecar scaffold skill** (`/cce:param-scaffold`) — Generates `.params.ts` sidecar files and `Tunable` wrapper boilerplate from component name + param list
10. **Work skill Astro/Svelte awareness** — Add `npm run dev`, tuning endpoint verification, and `npm run build` to the work loop's verification steps

### Implementation Location

All edits go to `~/claude-plugins/cce-marketplace/plugins/cce/`. Each batch is a self-contained unit of work — complete one batch, reinstall (`claude plugin uninstall cce@cce-marketplace && claude plugin install cce@cce-marketplace`), and verify in a new session before starting the next.

### Agent vs Augmentation Decision

- Zod discipline: **augment** `kieran-typescript-reviewer` (low effort, naturally integrated)
- Everything else: **new agents or skills** (distinct enough to warrant separation)
